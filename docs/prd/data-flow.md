# Excel 데이터 가공 Flow (비개발자용)

> humax-excel-mcp가 Excel 파일을 어떻게 처리해서 결산 산출물을 만드는지, 단계별로 그림과 비유로 설명. 상세 코드는 `src/humax_excel_mcp/`.

## 한 줄 요약

**raw 전표 데이터 → 월별 합계로 가공 → 골든 템플릿에 채우기 → 검증·백업까지 자동.** 사람이 손으로 하던 결산을 Python이 결정론(같은 입력 = 항상 같은 출력)으로 대신 수행.

## 전체 그림

```
   [입력]                  [가공]                    [출력]

  ┌────────┐   읽기    ┌─────────┐   채우기    ┌────────┐
  │ raw    │ ────────> │ 표 형태  │ ─────────> │ 산출물  │
  │ 전표   │           │ 데이터  │            │ 엑셀    │
  │ 엑셀   │           │ (메모리)│            │        │
  └────────┘           └─────────┘            └────────┘
                            │                      │
                            ▼                      ▼
                       검증·변환             백업·재확인
```

## 단계 1: 엑셀 열기 (Read)

**비유**: 도서관에서 책을 꺼내듯, 파일을 메모리에 불러온다.

- 파일이 잠겨 있으면 (다른 사람이 Excel로 열어둠) → 친절한 메시지: "Excel을 닫고 다시 시도하세요"
- xlsx가 아니면 거부
- 큰 파일(1.5만 줄 이상)은 **읽기 전용 모드**로 열어 메모리 절약

**중요 옵션 2개**:
- 읽을 때: 수식 셀의 **계산된 값**을 가져옴 (예: `=SUM(A1:A10)` → 1,250,000)
- 쓸 때: 수식 **문자열**을 그대로 보존 (덮어쓰지 않도록)

## 단계 2: 시트 → 표(DataFrame)

**비유**: 종이 장부를 Excel에 옮겨 적는 작업의 디지털 버전. 시트의 격자를 컴퓨터가 다루기 쉬운 표로 변환.

### 헤더(열 이름) 찾기

raw 데이터는 헤더가 1행이 아닐 수 있음 (제목·설명이 위에 있을 수 있음).

- **집계 시트**: 항상 1행이 헤더
- **raw 전표 시트**: 1행부터 10행까지 스캔 → 5개 이상 컬럼이 표준 이름과 일치하는 행을 헤더로 인정

### 헤더 정규화

한글 헤더를 시스템이 알아들을 수 있는 이름으로 변환:

| 원본 헤더 | 시스템 이름 |
|---|---|
| 회사 | `company` |
| Cost Center | `cost_center` |
| 1월 | `m01_actual` |
| 연간 예산 | `annual_budget` |

같은 이름이 두 번 나오면 `__1`, `__2` 자동 부여.

## 단계 3: 형식 감지 (raw냐 집계냐)

파일이 어떤 종류인지 자동 판정:

| 형식 | 모양 | 예시 |
|---|---|---|
| **raw 전표** | 한 줄 = 거래 1건. 날짜·회사·계정·금액 1줄씩 | 1월 인사팀 급여 5,000,000원 |
| **집계** | 한 줄 = 조직×계정 1조합. 12개월치 가로 | 인사팀 급여: 1월 500만, 2월 480만, ... |

**시스템이 첫 줄을 보고 자동 판정**. 못 알아보면 "스키마 불일치" 안내.

## 단계 4: raw → 집계 변환 (핵심 가공)

**여기가 가장 중요한 단계**. 전표 수천~수만 줄을 결산용 표로 압축.

### 4.1 안전 처리

- **개인정보 컬럼 즉시 삭제**: 적요 텍스트, 거래처명, URL, 전표번호 → LLM에 절대 전달 안 됨

### 4.2 필터링

```
모든 전표
  └── "26년"만 남김         (다른 연도 제외)
       └── "1월"~"12월" 파싱  (텍스트 → 숫자)
            └── target_month까지만   (예: 5월 결산이면 1-5월만)
```

### 4.3 그룹화 + 피벗 (Excel 피벗테이블과 동일 개념)

raw 데이터를 **(회사, 코스트센터, 계정과목)** 단위로 묶고, 월별로 가로로 펼침:

**Before (raw)**:
```
회사  CC      계정     구분  월   금액
A    인사    급여     예산  1월  5,000,000
A    인사    급여     실적  1월  4,800,000
A    인사    급여     예산  2월  5,000,000
A    인사    급여     실적  2월  4,950,000
...
```

**After (집계)**:
```
회사 CC   계정  m01_budget  m01_actual  m02_budget  m02_actual ...
A   인사 급여  5,000,000   4,800,000   5,000,000   4,950,000  ...
```

### 4.4 누계·연간 계산

월별 컬럼이 생기면 **자동 누계 + 연간 합산**:

- `cum01_actual` = 1월 실적
- `cum02_actual` = 1월 + 2월 실적
- `cum05_actual` = 1~5월 누적 실적
- `annual_budget` = `cum12_budget` (연간 예산 = 12개월 누계 = 연 예산)
- `annual_actual` = `cum{target}_actual` (예: 5월 결산이면 `cum05_actual`)

### 4.5 보강 컬럼 채우기

`계정명`, `조직명`, `배부율` 같은 부가 정보는 그룹의 첫 값을 그대로 가져와 채움.

### 4.6 EVCS 특수 처리 (`expand_evcs=True`)

전기차 충전 사업은 한 전표가 **국내/해외로 분할**됨. 시스템이 자동으로 가상 행 2개로 쪼갬:

```
원본 1줄:  공통경비 1,000,000 (국내 60%, 해외 40%)
              ▼
가상 2줄:
  EVCS국내  600,000
  EVCS해외  400,000
```

## 단계 5: 도구별 가공

집계된 표 위에서 도구가 각자의 일을 수행:

### `extract_filtered` — 표 필터링

- 월/회사/조직/계정그룹 조건으로 행 추리기
- **차이(variance)** 자동 계산: 실적 - 예산
- 차이 큰 순으로 정렬 → 상위 N건만 반환 (응답 크기 제한)

### `verify_sums` — 5계층 합계 검증

조직 계층(총합계 → 사업부 → 대조직 → 중조직 → 소조직)별로 **소조직 합 = 상위 합**인지 확인.

- 차이가 허용 범위(`tolerance`) 초과 → `FAIL`
- 1,000만원 이상 차이나는 행 → **이상치(anomaly)** 자동 추출 + 적요 초안 생성
- 합계 셀에 수식이 들어가 있어야 하는데 **숫자로 덮어쓰여진 경우** → 경고

### `generate_diff_candidates` — 전월 대비 변동

전월 파일과 당월 파일 비교:

```
(회사, CC, 계정) 공통 키로 매칭
   └── 차이 = 당월 실적 - 전월 실적
        └── |차이| ≥ 1,000만원 행만 후보
             └── 변동률 ≥ 50% → HIGH 심각도
                          ≥ 20% → MEDIUM
                          else  → LOW
                  └── 적요 초안 자동 작성
                       예: "51 급여 +30백만 (대폭 증가, +12.5%)"
```

### `get_allocation_rates` — 배부율 조회

26BP의 C30(배부기준) + C31~C34(배부율 4개) 읽고 **합이 100% 되는지** 검증. 99.9% → OK, 99.5% → 위반.

### `update_allocation_rates` — 배부율 변경

새 배부율 입력 → **합 100% 검증** → 헤더에서 컬럼 위치 자동 찾아 셀 수정.

### `apply_golden_template` — 골든 템플릿 채우기 ⭐

**핵심 안전 기능**: 매번 디자인이 달라지지 않도록 **빈 템플릿(서식+수식만)**에 데이터만 주입.

```
[빈 템플릿]              [집계 데이터]            [완성]
┌──────────────┐                                ┌──────────────┐
│ 회사 CC 금액  │                                │ 회사 CC 금액  │
│ ┌─┐  ┌─┐ ┌─┐│   ← 데이터 한 칸씩 주입 →     │A   인사 480만│
│ │ │  │ │ │ ││                                │A   재무 320만│
│ └─┘  └─┘ └─┘│                                │...           │
│ 합계: =SUM() │                                │ 합계: =SUM() │← 수식 보존
└──────────────┘                                └──────────────┘
```

**4가지 보호 장치**:
1. 병합 셀(merged cell)은 건너뜀
2. 수식 셀(`=SUM(...)`)은 건너뜀 → 사용자가 만든 수식 그대로
3. 데이터 셀만 정확한 좌표에 채움
4. 채운 후 첫 행을 다시 읽어 값 일치 확인 (spot-check)

### `generate_report` — 한 번에 산출물 생성

`apply_golden_template` + `verify_sums`를 묶어 자동 실행.

## 단계 6: 쓰기 (Write) — 항상 안전 우선

**비유**: 중요 문서 수정할 때 백업 사진 먼저 찍고 시작.

```
[1] 원본 == 출력 경로?     → 거부 (원본 직접 덮어쓰기 금지!)
       │
       ▼
[2] dry_run 모드?          → 시뮬레이션만, 파일 안 만듦
       │
       ▼
[3] 자동 백업              → .backup/원본_20260526_153200.xlsx
       │
       ▼
[4] sha256 지문 검증       → 백업이 원본과 100% 동일한지 확인
       │
       ▼
[5] 셀 수정 (수식 보호)    → "=" 시작 셀은 skip
       │
       ▼
[6] 저장
```

## 단계 7: 사후 검증 (Post-Write)

**비유**: 편지 보낸 후 받은 사람에게 "정확히 받았어요?"라고 다시 확인.

쓰기가 끝나도 안심하지 않음. 결과 파일을 **다시 열어서** 셀 값이 의도대로 들어갔는지 비교:

- 의도한 값 ≠ 실제 값 → "쓰기 후 검증 실패. 백업에서 복구하세요" 알림 + 에러
- 모두 일치 → OK

## 단계 8: 복구 (Restore)

문제가 생기면 백업에서 복구:

- 백업 파일 sha256 지문 계산
- 원본 위치에 복사
- 복사된 파일 지문이 백업 지문과 일치하는지 확인
- 불일치 → 복구 실패

**원본 자리에 직접 덮어쓰는 in-place 복구**는 한 번 더 확인 요구 (`confirm_overwrite_original=True`) + 복구 직전 현재 원본을 한 번 더 백업.

## Python 처리 vs LLM 직접 처리 비교

**핵심 원칙**: 숫자·산술·결정론은 Python, 자연어 이해·의도 파악은 LLM. 역할 분리가 안전·정확의 핵심.

### 왜 LLM이 직접 처리하면 안 되나?

| 문제 | LLM 직접 처리 시 | Python 처리 시 |
|---|---|---|
| **재현성** | 같은 입력에도 답이 미세하게 다를 수 있음 | 같은 입력 = **항상 같은 출력** (결정론) |
| **산술 정확도** | 큰 숫자 곱셈/누계에서 오차 가능 | 1원 단위 정확 |
| **속도** | 1.5만 행 처리 시 수 분 + 토큰 폭증 | 초 단위 |
| **비용** | 입력+출력 토큰 모두 과금 | API 호출 없음 |
| **개인정보** | raw 적요·거래처명·전표번호 LLM에 노출 | 가공 전 자동 삭제 |
| **감사 추적** | 어떤 계산을 했는지 불투명 | 코드 + JSONL 로그로 100% 재현 |
| **수식 보존** | "이 셀은 수식이야" 인식 어려움 | `=`로 시작하면 무조건 skip |

### 실제 사례

**❌ LLM 직접 처리하면 이런 사고**:

```
사용자: "1-5월 인사팀 급여 누계 계산해줘"
LLM:    "총 24,847,000원입니다"
실제:   24,850,000원 (3,000원 오차)
        → 결산 합계가 안 맞아 다시 작업
```

**✅ MCP 통한 Python 처리**:

```
사용자:  "1-5월 인사팀 급여 누계 계산해줘"
LLM:     extract_filtered(month="2026-05", company="인사팀", ...) 호출
Python:  raw → groupby → cumsum → cum05_actual 추출
결과:    24,850,000원 (정확)
LLM:     "1-5월 누계 24,850,000원입니다 (출처: cum05_actual)"
```

### 역할 분담 표

| 작업 | 담당 | 이유 |
|---|---|---|
| 사용자 의도 파악 | **LLM** | "5월 결산 만들어줘" → 어떤 도구·파라미터인지 판단 |
| 자연어 → 도구 호출 | **LLM** | 회사명·월·계정 파라미터 추출 |
| 파일 읽기 | **Python** | openpyxl 라이브러리 |
| 헤더 한글 → 정규화 | **Python** | 미리 정의된 매핑 테이블 |
| 필터링 | **Python** | pandas DataFrame |
| 월별 피벗 | **Python** | pandas pivot_table |
| 누계 계산 | **Python** | numpy cumsum |
| 차이·변동률 계산 | **Python** | 단순 산술 |
| 100% 합계 검증 | **Python** | tolerance 기반 비교 |
| 백업·sha256 | **Python** | hashlib |
| 셀 좌표 쓰기 | **Python** | openpyxl |
| 결과 해석·보고 | **LLM** | "이상치 3건 발견, 가장 큰 차이는..." |
| 적요 초안 작성 | **Python (템플릿)** | `"{CODE} {계정} {±N백만}"` 고정 포맷 |
| 적요 문맥 보강 | **LLM** | 사용자 컨텍스트 보고 자연스럽게 다듬음 |

### Python에 맡기는 결정적 이유 4가지

**1. 결정론 (Determinism)**

같은 raw 파일을 100번 처리해도 결과는 100번 모두 동일. 회계 결산에서 필수.

**2. 감사 추적 (Auditability)**

모든 도구 호출은 JSONL 로그에 기록 (`.humax-mcp/audit/`). "누가 언제 어떤 파라미터로 무엇을 호출했고 결과는 어땠는지" 100% 재구성 가능. LLM 내부 계산은 추적 불가.

**3. 안전 정책 강제 (Safety)**

원본 덮어쓰기 차단, 백업 강제, sha256 검증, dry-run, 수식 보호 — **코드로 보장**. LLM에게 "조심해줘"라고 부탁하는 게 아니라 시스템이 거부.

**4. 비용·속도**

1.5만 행 raw 데이터 가공:
- Python: **2-5초** (메모리 내 pandas pivot)
- LLM 직접: **수 분 + 수십만 토큰** (재시도 시 비용 폭증)

### LLM이 핵심 역할 하는 영역

Python이 못 하는 부분은 LLM이 담당:

- **"5월 결산 자료 만들어줘"** → 어떤 도구 조합인지 추론
- **"제미니 사업부 인건비 변동 봐줘"** → 회사명·조직·계정 파라미터로 변환
- **이상치 결과 받고** → "급여 +30백만은 4월 신입사원 채용 영향으로 추정됩니다" 등 컨텍스트 추론
- **적요 자연스럽게 다듬기** → 템플릿 초안 → 사용자 회사 톤에 맞게 윤문
- **에러 메시지 친절히 안내** → "Cost Center 컬럼이 없습니다" → "엑셀의 코스트센터 컬럼명을 확인해주세요"

### 한 줄 정리

> **숫자는 Python, 말은 LLM.** Python이 결정론·안전·속도·비용을 보장하고, LLM이 자연어 이해·의도 파악·문맥 보강을 담당. 둘 다 자기 잘하는 일만 한다.

## 안전·결정론 보장 5가지

| 보장 | 무엇 | 왜 |
|---|---|---|
| **결정론** | 모든 계산은 Python (pandas/numpy) | LLM이 숫자 만들지 않음 → 같은 입력 = 같은 출력 |
| **백업 always** | write 도구는 무조건 백업 먼저 | 실수 시 즉시 복구 가능 |
| **sha256 지문** | 백업·복구 파일 무결성 | 1바이트라도 다르면 즉시 감지 |
| **post-write 재확인** | 쓰고 나서 다시 읽어 비교 | 저장 중 손상 감지 |
| **dry-run** | 실제 안 쓰고 시뮬레이션만 | 미리 결과 확인 가능 |

## 자주 묻는 질문 (FAQ)

**Q. 원본 파일이 손상될 수 있나?**
A. 시스템이 원본 직접 덮어쓰기를 **거부**합니다. 출력은 항상 다른 파일로 저장되고, 백업까지 자동 생성됩니다.

**Q. 수식이 사라지나?**
A. 시스템이 `=`로 시작하는 셀을 **자동 건너뜀**. 사용자가 만든 SUM·VLOOKUP 등은 그대로 보존됩니다. (`skip_if_formula=False`로 명시할 때만 덮어씀 + 경고)

**Q. LLM이 계산을 잘못하면?**
A. 모든 산술은 Python (pandas/numpy)이 수행. LLM은 입력 파라미터만 만들고 결과를 받아 보고할 뿐, 숫자에 손대지 않습니다.

**Q. 큰 파일 처리되나?**
A. 1.5만 줄+ 파일도 처리. **읽기 전용 모드**로 메모리 절약, **수식 평가 단계 스킵**으로 시간 단축 (분 단위 → 초 단위).

**Q. 개인정보가 새지 않나?**
A. raw 데이터에 있는 **적요·거래처명·URL·전표번호**는 집계 변환 진입 직전 자동 삭제. LLM 응답에 절대 포함되지 않습니다.
