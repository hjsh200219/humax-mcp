# core-beliefs.md

> 에이전트 우선 운영 핵심 신념.

## 1. 결정론 > LLM 산술

LLM은 산술/집계/검증을 비결정적으로 수행한다. 같은 입력에 다른 출력이 나올 수 있다. 따라서 모든 산술은 Python으로 위임하고, LLM은 자연어 → 도구 호출 매핑만 담당한다.

근거: file #1 에러 #5 "작업 후 숫자 검토작업 필요" — LLM 확률적 생성이 원인.

## 2. 원본 보존 > 편의

write 도구는 원본을 절대 덮어쓰지 않는다. 모든 mutate에 자동 백업 + sha256 검증을 강제한다. 사용자가 명시하지 않은 변경은 발생하지 않는다.

근거: file #1 에러 #1 "원본파일 훼손, 복구불가" + 에러 #2 "지시하지 않은 내용 변경".

## 3. 코드 = 기록 시스템

레이어 구조, 의존 방향, 에러 코드, 도구 등록 — 모두 코드에 명시적으로 인코딩한다. 문서는 코드를 설명하는 맵일 뿐이며, 코드와 문서가 불일치하면 코드가 정답이다.

근거: 하네스 엔지니어링 원칙 "리포지터리 = 기록 시스템".

## 4. Search Before Building

신규 helper 작성 전 `core/` `schemas/` `tools/`의 기존 모듈을 grep한다. 동일한 백업/감사/스키마 검증 로직을 중복으로 작성하지 않는다.

근거: 하네스 엔지니어링 원칙 "Search Before Building".

## 5. TDD 우선

새 기능/로직 변경 시 실패하는 테스트를 먼저 작성한다 (Red → Green → Refactor). 234 테스트의 일관된 패턴은 이 신념에서 비롯됐다.

근거: progress.txt — Ralph TDD 세션 14 US 전부 TDD로 완성.

## 6. 회의적 자기 평가

자기 평가 편향을 상쇄하기 위해 verifier는 "증명될 때까지 미흡" 관점으로 본다. "테스트가 통과한다"는 사실은 "회귀를 잡는다"의 증명이 아니다. 커버리지 게이트, E2E 체인, sha256 검증으로 보강한다.

근거: Anthropic GAN-inspired evaluator pattern.

## 7. 실무자 어휘로 설계

도구명/파라미터/에러 메시지는 실무자 어휘를 사용한다. `extract_filtered` 도구의 `sheet_name`, `companies`, `months`는 영어지만 의미가 직관적. 에러 메시지는 한국어 통일.

근거: 사용자 페르소나 — Humax 재무 실무자, CLI 미경험.

## 8. 점진적 공개

진입 문서 (AGENTS.md)는 ~100줄 맵. 상세는 docs/ 하위 링크로 점진 공개. 에이전트가 작고 안정적인 진입점에서 시작해 필요한 만큼만 깊이 탐색한다.

근거: 하네스 엔지니어링 원칙 "Map, not handbook".
